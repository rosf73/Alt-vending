> 상위 규범: [spec.md](../spec.md)
> 대응 화면 요구사항: [requirements/admin-screen.md](../requirements/admin-screen.md)

## B-1. 목적

운영자가 자판기의 **재고, 잔돈, 매출**을 관리하고, 향후 **감사 로그**를 확인할 수 있도록 하는
관리 기능의 비즈니스 규칙을 정의한다.

---

## B-2. 하위 기능

| 기능 | 상태 | 설명 |
|------|------|------|
| B-2.1 재고 관리 | 필수 | 상품 목록/가격/수량 조회 및 변경 |
| B-2.2 잔돈 관리 | 필수 | 자판기 보유 잔돈(화폐별 개수) 조회 및 보충/회수 |
| B-2.3 매출액 확인 | 필수 | 누적 매출액 및 상품별 판매 현황 조회 |
| B-2.4 감사 로그 | **구현 완료** | 관리자 조작·판매 이벤트 기록 조회 (백엔드 영속) |

## B-3. 기능 규칙

### B-3.1 재고 관리
- 조회: 상품별 `상품명, 기본가, 수량` 목록.
- 변경 가능 항목:
  - 수량 보충/차감 (결과 값 0 미만 불가 — INV-6)
  - **기본가** 변경 (**100 ~ 99,999원**, 범위 밖 입력 거부 — BR-B6)
  - 상품 추가/제거 (**최대 9개**까지, 9개 상태에서 추가 거부 — BR-B10). 판매 화면 **3×3 그리드**(sales-mode A-3.1)와 슬롯 수가 일치. 최소한 예제 4종은 시드.
- 변경은 즉시 판매 모드의 상품 디스플레이에 반영된다(상태 단일 출처).
- > 관리자는 **기본가**를 관리한다. 판매 화면에 노출/청구되는 **최종가**는 동적 가격(+500)·프로모 할인(−500)이 자동 반영된 파생값이다(sales-mode A-7.2). 재고를 3개 이하로 두면 동적 인상이 발동함에 유의.

### B-3.2 잔돈 관리
- 조회: 화폐 단위별(1,000 / 500 / 100) 보유 **개수** 및 환산 총액.
- **보충**: 특정 단위의 개수를 증가.
- **회수**: 특정 단위의 개수를 감소(결과 0 미만 불가).
- 잔돈 보유량은 판매 모드의 "거스름돈 반환 가능 여부"(BR-A4)에 직접 영향을 준다.

### B-3.3 매출액 확인
- 누적 매출 총액 조회.
- (선택) 상품별 판매 수량/금액 조회.
- 매출은 **배출 성공 시(BR-A5)에만** 최종가만큼 증가하며, 반환/자동반환으로는 변하지 않는다.
- **배출 실패 미반환금**(특별 BM A-7.3)은 매출이 아니라 **별도 항목**으로 집계한다(INV-4 원장 보존). (선택) 관리자 화면에 매출과 구분해 표시.
- (선택) 매출 초기화(정산) 기능 — 실행 시 감사 로그에 기록 대상.

### B-3.4 감사 로그 (구현 완료 — 2026-07-02 고도화)
- 기록 대상 이벤트(전부 구현):
  - 판매 성공/실패, 자동 반환, 수동 반환
  - 재고 변경, 가격 변경
  - 잔돈 보충/회수
  - 상품 추가/제거, 매출 초기화, 모드 전환
- 각 항목: `시각(ISO), 이벤트 유형, 상세(변경 전/후 값), 결과(OK/FAIL)`.
- **적재**: 백엔드 SQLite `audit_log` 테이블에 **append-only** 영속(INV-1). 실패한 관리자 조작도 결과 `FAIL`로 남긴다.
- **조회**: `GET /api/audit`로 최신순 조회, 관리자 화면 "감사 로그" 탭에 표시(REQ-B11). 상태 변경 이벤트 발생 시 실시간 갱신.
- **구현 위치**: 타입 `domain/audit.ts`, 적재/조회 `persistence/store.ts`(`appendAudit`·`listAudit`), 이벤트 기록 `server/service.ts`.
- 전체 초기화(`reset`) 시 `audit_log`도 비운다.

## B-4. 모드 전환 규칙

| ID | 규칙 |
|----|------|
| BR-B1 | 관리자 모드와 판매 모드는 **동시에 열릴 수 있다**(다른 탭). 상호 배타가 아니다(INV-7 개정). |
| BR-B2 | 관리자 진입은 진행 중인 판매 세션을 **중단시키지 않는다**. 진입 시 강제 반환 없음. 관리자 변경은 진행 중 세션에도 실시간 반영된다. |
| BR-B3 | 관리자 변경 사항은 영속 저장되어 재시작/새로고침 후에도 유지된다(INV-1). |
| BR-B11 | 모든 관리자 변경(재고·가격·잔돈)은 **연결된 모든 판매 클라이언트(탭)에 실시간(SSE) push**되어 즉시 반영된다(INV-7). |

> 관리자 인증(비밀번호/키) 방식은 요구사항 미명시. 실제 자판기 관례에 따라 **간단한 진입 게이트**를 두는 것으로 가정(헌법 §3.4).

## B-5. 비즈니스 규칙 표 (Rule Table)

| ID | 규칙 | 결과 |
|----|------|------|
| BR-B4 | 재고 수량을 음수로 만드는 변경 | 거부 |
| BR-B5 | 잔돈 개수를 음수로 만드는 회수 | 거부 |
| BR-B6 | 기본가를 **100~99,999원 범위 밖**으로 설정 | 거부 (경계 100·99,999은 허용) |
| BR-B7 | 재고/가격 변경 | **모든 판매 탭에 실시간(SSE) 반영**(INV-7) — 최종가·품절·색상 재평가 |
| BR-B8 | 잔돈 보충 | 판매 모드 거스름돈 가용성 증가 |
| BR-B9 | 모든 관리자 변경 | 영속 저장 + 감사 로그 적재(구현 완료, B-3.4) |
| BR-B10 | 상품이 **9개인 상태에서 추가** | 거부 (최대 9개, 3×3 그리드) |

## B-6. 예시 (Worked Examples)

- **잔돈 보충**: 100원 개수 10 → 50으로 변경 → 총액 +4,000원 → 이후 소액 거스름돈 반환 여유 증가.
- **품절 해소**: 콜라 수량 0 → 5로 보충 → 판매 모드에서 "품절" → "구매 가능/금액 부족"으로 전환.
- **매출 확인**: 콜라 3, 커피 2 판매 → 누적 매출 = 1,500×3 + 1,800×2 = 8,100원.

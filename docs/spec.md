## 1. 프로젝트 개요

실제 음료 자판기의 동작을 재현하는 **자판기 시뮬레이터**를 구현한다.
비즈니스 로직과 상태를 관리하는 **백엔드**와, 사용자 상호작용을 담당하는
**프론트엔드(모던 웹브라우저)**로 구성한다.

- **판매 모드(Sales Mode):** 일반 사용자가 음료를 구매.
- **관리자 모드(Administrator Mode):** 운영자가 재고·잔돈·매출을 관리.
- 두 모드는 **서로 다른 탭에서 동시에** 열릴 수 있고, 상태 변경은 모든 클라이언트에 실시간 동기화된다(INV-7 개정).

## 2. 불변 원칙 (Invariants)

프로젝트 전 기간 동안 반드시 지켜져야 하는 규칙. 위반 시 버그로 간주한다.

| # | 원칙 | 근거 |
|---|------|------|
| INV-1 | **영속성**: 백엔드 재시작 또는 페이지 새로고침 후에도 재고·잔돈·매출·투입금액 등 모든 상태가 유지된다. | 요구사항 명시 |
| INV-2 | **단일 상태·다중 클라이언트**(개정): 자판기는 1대(백엔드 **단일 공유 상태**)다. 단, **여러 브라우저 탭이 동시에 접속·조작**할 수 있으며, 상태를 변경하는 모든 연산은 백엔드에서 **직렬화(원자적)**된다. | 요구사항 명시 + 동시성 피드백 |
| INV-3 | **화폐 제한**: 100원·500원·1,000원만 투입/처리한다. 그 외 화폐는 거부한다. | 요구사항 명시 |
| INV-4 | **금액 원장 보존**(개정): 기계 관점에서 돈은 생성/소멸하지 않는다 — `투입 총액 = 반환 총액 + 매출 + 배출 실패 미반환금`. 정상 거래에서는 사용 금액이 매출로, 초과분이 반환으로 귀속되나, **배출 실패 시(특별 BM §3.5·sales-mode A-7.3)** 사용 금액은 사용자에게 반환되지 않고 기계가 "미반환금"으로 보유한다. | 도메인 정합성 / BM |
| INV-5 | **최소 개수 잔돈**: 잔돈은 항상 **가능한 최소 개수의 화폐**로 반환한다. | 요구사항 명시 |
| INV-6 | **재고/잔돈 음수 불가**: 재고 수량과 잔돈 보유량은 0 미만이 될 수 없다. | 도메인 정합성 |
| INV-7 | **모드 공존·실시간 동기화**(개정): 판매 모드와 관리자 모드는 서로 다른 탭에서 **동시에 열릴 수 있다**. 상태 변경(특히 **관리자 수정**)은 모든 클라이언트에 **실시간 반영**된다. | 동시성 피드백 |

## 3. 도메인 규칙 (Domain Rules)

### 3.1 취급 화폐
- 투입 가능: **100원, 500원, 1,000원** (지폐/동전 구분 없이 액면가로 취급)
- 잔돈 반환 단위: **1,000원, 500원, 100원**
- 잔돈 계산: 큰 단위부터 소진하는 **탐욕적(greedy) 최소 개수** 방식.
  단, 자판기 보유 잔돈이 부족하면 반환 불가 상황이 발생할 수 있다(→ 예외 정책, 3.4).

### 3.2 예제 상품 (초기 데이터)
| 상품 | 가격 | 비고 |
|------|------|------|
| 콜라 | 1,500원 | |
| 사이다 | 1,200원 | |
| 생수 | 500원 | |
| 커피 | 1,800원 | |

> 상품 목록·가격·수량은 관리자 모드에서 변경 가능한 **데이터**이며, 위 값은 초기 시드값이다.

**상품 제약(도메인 규칙):**
- **최대 상품 개수 = 9** (판매 화면 3×3 그리드와 일치). 초과 추가 불가.
- **기본가 범위 = 100 ~ 99,999원** (경계 포함). 범위 밖 설정 불가.
- 상세/검증은 [business/admin-mode.md](business/admin-mode.md) B-3.1·BR-B6·BR-B10.

### 3.3 타이밍 규칙
| 규칙 | 값 | 설명 |
|------|----|------|
| 상품 배출 지연 | **1~2초** | 선택·결제 확정 후 배출 완료까지의 지연 |
| 미선택 자동 반환 | **10초** | 현금 투입 후 상품 미선택 상태로 10초 경과 시 투입금 전액 자동 반환 |

### 3.4 예외/경계 정책
- **잔액 부족**: 투입 금액이 상품 가격보다 적으면 해당 상품은 "구매 불가"로 표시하고 선택을 거부한다.
- **재고 없음(품절)**: 수량 0인 상품은 "품절"로 표시하고 선택을 거부한다.
- **거스름돈 부족**: 최소 개수로도 정확한 잔돈을 만들 수 없으면(자판기 잔돈 부족) 구매를 거부하고 투입금을 반환한다.
- **자동 반환 타이머**: 사용자가 화폐를 추가 투입하면 10초 타이머를 리셋한다(실제 자판기 관례 기준).

> "기타 미명시 사항은 실제 음료 자판기의 일반적 동작을 기준으로 구현한다." (요구사항 원문)

### 3.5 특별 비즈니스 규칙 (Special BM) — 요약
> 상세 로직은 [business/sales-mode.md](business/sales-mode.md) A-7, 세션 모델은 A-2. 명시되지 않은 세부 해석은 sales-mode A-6(가정)에 기록되며 일부는 **확정 필요**.

- **세션**: 현금 투입으로 시작, **잔액이 0이 되면 종료**(정확 소진 또는 반환). 세션 내 **다회 구매** 가능하며, 거스름돈은 **세션 종료 시 일괄 반환**한다.
- **프로모 할인**: 세션 내 **3·6·9…번째 구매 상품 −500원**(2번 구매마다 3번째 할인).
- **동적 가격**: **재고 3개 이하인 그 상품만** 가격 **+500원**(전역 아님). 인상된 최종가는 **붉은색**, 평상시 최종가는 **노란색**으로 표시(design-system §5.1).
- **배출 실패**: 배출은 **90% 성공 / 10% 실패**, 실패 시 사용 금액 **미반환**(INV-4 개정 반영).
- **최종가** = `max(0, 기본가 + 동적 인상 − 프로모 할인)`.

## 4. 용어 정의 (Glossary)

| 용어 | 정의 |
|------|------|
| 투입 금액 (Inserted Amount) | 사용자가 투입한 화폐의 합계. **잔액**으로 누적된다 |
| 잔액 (Balance) | 세션 내 현재 사용 가능한 누적 금액. 구매 시 최종가만 차감되고, 거스름돈은 세션 종료 시 반환 |
| 최종가 (Effective Price) | 기본가에 동적 인상(+500)·프로모 할인(−500)을 반영한 실제 청구 금액 (§3.5) |
| 잔돈 (Change) | **세션 종료 시** 반환하는 잔여 잔액 / 자판기가 반환용으로 보유한 화폐 |
| 재고 (Inventory) | 상품별 보유 수량 |
| 매출액 (Sales / Revenue) | 판매(배출 성공)로 누적된 금액 합계 |
| 배출 실패 미반환금 | 배출 실패 시 기계가 보유하게 되는 미반환 금액(매출 아님, INV-4) |
| 감사 로그 (Audit Log) | 관리자 조작·판매 이벤트의 기록 (**구현 예정**, planned) |
| 배출 (Dispense) | 상품을 사용자에게 내보내는 동작 (특별 BM상 90% 성공) |
| 자동 반환 (Auto-refund) | 미구매 타임아웃 시 잔여 잔액 반환 |
| 세션 (Session) | 현금 투입으로 시작, **잔액이 0이 되면 종료**되는 거래 단위. 세션 내 다회 구매 가능 (§3.5·A-2) |

## 5. 아키텍처 대원칙

1. **관심사 분리**: 비즈니스 로직(백엔드)은 UI와 독립적이어야 하며, UI 없이도 테스트 가능해야 한다.
2. **상태의 단일 출처(Single Source of Truth)**: 재고·잔돈·매출 등 진실 데이터는 백엔드에 있으며, 프론트엔드는 이를 표시/요청만 한다.
3. **영속 저장소**: 상태는 프로세스 메모리가 아닌 영속 저장소(DB/파일 등)에 저장하여 INV-1을 보장한다.
4. **원자성**: "결제 → 재고 차감 → 잔돈 차감 → 매출 증가"는 하나의 원자적 트랜잭션으로 처리한다(부분 실패 금지).
5. **동시성 직렬화**: 여러 클라이언트(탭)가 동시에 조작해도, 상태 변경 연산은 백엔드에서 **직렬화**되어 경합/유실이 없도록 한다(INV-2).
6. **실시간 동기화**: 상태 변경은 백엔드가 **모든 연결된 클라이언트에 실시간 push**한다(관리자 수정 → 판매 화면 즉시 반영, INV-7). 전송은 §8.1의 실시간 채널을 사용한다.

## 6. 범위 (Scope)

### 포함
- 판매 모드 / 관리자 모드 전체 기능
- 상태 영속성, 최소 개수 잔돈, 자동 반환, 배출 지연
- 특별 BM: 세션 다회 구매, 프로모 할인, 재고 기반 동적 가격, 배출 실패(§3.5)

### 이번 범위 밖 (Out of Scope)
- 감사 로그 UI/조회 (기능은 **예정**으로 명시, 데이터 적재 설계만 고려)
- 다중 자판기 / 사용자별 독립 세션 분리 (다중 탭은 **단일 공유 상태**를 함께 보는 것이며, 사용자마다 별도 상태를 두지는 않음 — INV-2)
- 결제 카드/간편결제 등 현금 외 수단

## 8. 기술 스택 & 플랫폼 (고정)

> **이 표가 스택의 단일 출처(Single Source of Truth)다.** 스택 변경은 이 표만 수정하고, 하위 코드는 이를 따른다.
> 선택 기준: **LLM 인지도가 높은 표준 조합**(→ 이탈 최소화), **타입 안전성**(→ 스펙 이탈을 컴파일 타임에 차단), **무설정 파일 영속성**(→ INV-1 자동 충족).

### 8.3.1 구현 아키텍처 결정 (Decision Record — R7)

> 2026-07-02, 사용자 승인. 근거: "앱은 무선랜 환경에서 심사위원의 브라우저로 접속 가능하도록 호스팅"해야 함.

- **채택:** **하이브리드 단일 프로젝트**(프로젝트 루트 단일 `package.json`). `backend/`·`frontend/` 물리 분리(§8.2) 대신 루트에 **co-locate**한다.
  - `domain/` — 순수 도메인 로직(UI/DB 무관, 100% 단위테스트 대상). §8.2의 `backend/src/domain` 역할.
  - `persistence/` — SQLite 리포지토리(INV-1).
  - `server/` — Express 라우트 + SSE, 정적 프론트 호스팅(심사위원 접속).
  - `index.ts` — **프론트엔드 진입점(UI)**. 같은 경로에 CSS/TS 모듈을 두고 import한다. Vite로 빌드/서브.
- **불변식 유지:** 스택(§8.1)·폴더 역할은 유지되며, 물리 배치만 루트 co-locate로 조정. INV-1(SQLite)·INV-7(SSE)·R9(도메인 단일 출처)는 그대로 준수한다.

### 8.3 플랫폼 제약
- **타깃:** 모던 웹브라우저(Chrome/Edge/Safari 최신).
- **뷰포트(고정 프레임·비하이브리드):** 앱은 하이브리드/반응형으로 폭을 확장하지 않고 **고정 프레임**으로 동작한다. **최대 width = 500px**(그 이상은 좌우 여백, 중앙 정렬), **height는 항상 뷰포트 최대치(100vh)** 유지. (모바일형 세로 프레임 — UI: design-system §7)
- **실행:** 로컬 단일 백엔드 프로세스(단일 공유 상태). **여러 브라우저 탭이 동시 접속**할 수 있으며, 상태 변경은 백엔드에서 직렬화되고 SSE로 실시간 전파된다(INV-2·INV-7). 다중 서버 수평 확장은 범위 밖(§6).

## 9. LLM 행동 규정 (LLM Operating Rules)

> 구현/수정을 수행하는 AI 에이전트가 **반드시(MUST)** 지키는 규범. 위반 = 이탈(drift)로 간주하고 §12로 보정한다.

| ID | 규정 | 세부 |
|----|------|------|
| **R1** | **스펙이 유일한 진실** | 구현은 헌법·`business/*`·`requirements/*`·`ui/*`에서만 도출한다. 문서에 없는 동작은 지어내지 않는다. |
| **R2** | **범위 침범 금지** | §6 Out of Scope(다중 자판기·현금 외 결제·감사로그 UI 등)를 구현하지 않는다. |
| **R3** | **불변식 최우선** | INV-1~7을 깨는 코드는 작성하지 않는다. 규칙 충돌 시 §7 우선순위를 따른다. |
| **R4** | **추적성 필수** | 모든 함수/모듈/커밋은 관련 `INV·REQ·BR·TC` ID를 주석 또는 커밋 메시지에 명시한다. (예: `// TC-A07, REQ-A5, INV-4`) |
| **R5** | **테스트 우선(TDD)** | §10 절차를 따른다. **실패하는 테스트 없이 프로덕션 코드를 추가하지 않는다.** |
| **R6** | **작은 단위로 진행** | 한 번에 하나의 TC(또는 REQ)만 다룬다. 대규모 일괄 변경 금지. |
| **R7** | **결정은 기록** | 문서에 없어 부득이 판단한 사항은, 임의로 코드에만 반영하지 말고 해당 spec 문서의 **"가정(Assumptions)"** 절에 먼저 적는다. |
| **R8** | **검증 없는 완료 선언 금지** | §11.4 게이트를 모두 통과하기 전에는 "완료"라고 말하지 않는다. |
| **R9** | **상태 단일 출처** | 도메인 상태·규칙은 `backend/domain`에만 둔다. 프론트엔드에 비즈니스 규칙(가격 계산·잔돈 로직 등)을 복제하지 않는다. |
| **R10** | **모호하면 멈춘다** | 스펙이 모호/누락이고 맥락이 없으면 §3.4의 "실제 자판기 관례"를 적용하되 그 결정을 R7대로 기록한다. 관례로도 판단 불가면 사람에게 질문한다(추측 금지). |

## 10. TDD 규율 (Test-Driven Discipline)

### 10.1 사이클
**RED → GREEN → REFACTOR** 를 엄격히 따른다.
1. **RED**: 요구사항의 `TC-*`를 표현하는 테스트를 먼저 쓴다 → 실행해 **실패**를 확인한다.
2. **GREEN**: 그 테스트를 통과시키는 **최소** 코드를 쓴다.
3. **REFACTOR**: 테스트가 green인 상태를 유지하며 중복 제거·네이밍 정리.

### 10.2 테스트 출처와 계층
- **1차 소스는 `requirements/*.md`의 TC 표다.** 각 `TC-*`는 최소 1개의 자동화 테스트로 구현된다.
- 계층 매핑:

| 대상 | 도구 | 무엇을 검증 |
|------|------|-------------|
| 도메인 순수 로직 | Vitest 단위 | INV-1~7, BR-A*/BR-B*, 최소개수 잔돈 등 |
| HTTP API | Supertest 통합 | 요청→상태변화→응답 계약 |
| 화면 흐름 | Playwright E2E | `TC-A*`(판매), `TC-B*`(관리자) |

### 10.3 규칙
- **모든 INV에는 대응 테스트가 존재해야 한다** (INV → 테스트 역추적 가능).
- 테스트 이름에 TC ID를 포함한다. 예: `test('TC-A09: 세션 종료 시 잔여 잔액 최소개수 반환', …)`.
- 커버리지 게이트: **`backend/src/domain` 라인 커버리지 ≥ 90%**.
- 발견된 버그/이탈은 반드시 **회귀 테스트**로 고정한 뒤 수정한다(§12.5).
- 커밋은 `red`(실패 테스트) → `green`(구현)을 분리하는 것을 권장한다.

## 11. 구현·검증 플로우 (START HERE)

> **이 절은 하네스의 본체다.** 이전 대화·맥락이 전혀 없는 LLM이라도, 이 절만 위에서 아래로 따르면 스펙을 벗어나지 않고 구현/검증을 완결할 수 있다.
> 각 단계는 **입력 → 행동 → 출력(통과 기준)**이 명확하며, 실패하면 항상 §11.5(보정 루프)로 간다.

### 11.0 한 줄 요약
```
스펙 읽기 → TC 하나 선택 → RED → GREEN → INV 점검 → REFACTOR → 게이트(G1~G7) → 통과? 다음 TC : 보정 루프(§11.5)
```

### 11.1 준비 (프로젝트당 1회)
1. 문서를 이 순서로 읽는다: **§1~§12(본 헌법) → `business/*` → `requirements/*` → `ui/*`**.
2. §8.1 스택으로 `backend/`·`frontend/`를 초기화하고 `npm install`.
3. §8.2 폴더 구조를 만든다. `npm test`가 (테스트 0개라도) 정상 실행되는지 확인.
4. 예제 상품·초기 잔돈 시드(§3.2)를 SQLite에 적재하는 스크립트를 둔다.

### 11.2 작업 단위 선택
- `requirements/sales-screen.md` 또는 `requirements/admin-screen.md`의 TC 표에서 **아직 통과하지 못한 TC를 하나** 고른다(작고 독립적인 것부터).
- 그 TC 행의 **매핑 컬럼(REQ/INV/BR)**을 확인한다. 이것이 "무엇을 만족시켜야 하는가"의 정의다.

### 11.3 구현 루프 (선택한 TC 1건 기준)

| 순서 | 행동 | 통과 기준 |
|------|------|-----------|
| 1. 이해 | TC의 `Given/When/Then`과 매핑된 REQ/BR/INV를 읽는다. | 기대 동작을 한 문장으로 말할 수 있다 |
| 2. **RED** | 해당 계층(§10.2)에 이 TC를 표현하는 테스트를 작성·실행. | 테스트가 **의도한 이유로 실패**한다 |
| 3. **GREEN** | 통과시키는 최소 코드 작성. 도메인 로직은 `backend/src/domain`에. | 그 테스트가 **통과**한다 |
| 4. INV 점검 | INV-1~7 위반 여부 검토(특히 원장 보존 INV-4, 음수 금지 INV-6). | 위반 없음, 필요 시 INV 테스트 추가 |
| 5. **REFACTOR** | 중복 제거·네이밍 정리. | 모든 테스트 green 유지 |
| 6. 추적성(R4) | 코드 주석·커밋에 `TC/REQ/INV` ID 기록. | ID가 코드에서 검색된다 |
| 7. 게이트 | §11.4의 G1~G7 실행. | 전부 통과 → §11.6 / 하나라도 실패 → §11.5 |

### 11.4 검증 게이트 (Definition of Done — 기계 검증 가능)

> 대상 TC가 "완료"로 인정되려면 **G1~G7을 모두 통과**해야 한다(R8).

| 게이트 | 명령(예시) | 통과 기준 | 근거 |
|--------|-----------|-----------|------|
| **G1 타입** | `npm run typecheck` | 타입 에러 0 | R3 |
| **G2 린트** | `npm run lint` | 에러 0 | 품질 |
| **G3 테스트** | `npm test` | 단위·통합 전부 pass | §10 |
| **G4 커버리지** | `npm run test:cov` | `domain` 라인 ≥90% & 모든 INV 대응 테스트 존재 | §10.3 |
| **G5 화면(E2E)** | `npm run e2e` | 대상 `TC-*` pass | §10.2 |
| **G6 영속성** | 백엔드 재시작 → 상태 재조회 | 재고·잔돈·매출이 유지됨 | **INV-1** |
| **G7 추적성** | `grep`로 변경분에 TC/REQ ID 확인 | ID 존재 | R4 |

### 11.5 실패 시 → §12 보정 루프로 이동
게이트가 하나라도 실패하면 그것은 **스펙 이탈(drift)**이다. **임의로 코드를 덧붙이지 말고** §12 절차로 원인을 분류·교정한 뒤 §11.4로 돌아온다.

### 11.6 완료 처리
- G1~G7 전부 통과 → 해당 TC를 "통과"로 간주하고 §11.2로 돌아가 다음 TC를 고른다.
- 모든 TC가 통과하고 각 requirements 문서의 "완료 기준(Acceptance)"이 충족되면 해당 화면/기능이 완료된다.

## 12. 이탈 감지·보정 하네스 (Drift Detection & Correction)

> **하네스의 목적:** 맥락 없이 진행된 구현이 스펙에서 어긋나더라도, 게이트가 이를 **잡아내고(감지)** 스펙 쪽으로 **되돌린다(보정)**. 추측으로 덮지 않는다.

### 12.1 이탈의 정의
아래 중 하나라도 발생하면 이탈이다.
- 게이트 G1~G7 중 하나라도 실패.
- INV-1~7 위반, 또는 §6 범위(Out of Scope) 침범(R2).
- 스펙에 근거 없는 동작이 코드에 존재(R1 위반) — 추적 ID 없음(R4 위반).

### 12.2 보정 절차 (Correction Loop)
1. **분류(Locate):** 어떤 게이트/규칙이 깨졌고, 어떤 스펙 ID(INV/REQ/BR/TC)와 어긋났는지 특정한다.
2. **원인 판별(Diagnose):**
   - **(a) 코드가 스펙과 다르다** → 코드를 스펙에 맞게 수정하고 테스트를 다시 green으로.
   - **(b) 스펙이 모호/누락이다** → 코드로 임의 결정하지 말고, §3.4 관례를 적용하되 그 결정을 해당 문서의 "가정" 절에 기록(R7). 관례로도 불가하면 멈추고 질문(R10).
   - **(c) 테스트가 스펙을 잘못 표현했다** → 테스트를 TC 정의에 맞게 고친다.
3. **교정(Fix):** 위 판별에 따라 최소 변경을 적용한다.
4. **회귀 고정(Pin):** 이탈을 재현하는 테스트를 남겨 재발을 막는다(§10.3).
5. **재검증(Re-verify):** §11.4 게이트를 다시 실행한다.
6. **정지 조건(Stop):** 같은 이탈로 **3회** 보정에 실패하면 추측을 멈추고, 이탈 내용·시도한 조치·의심되는 스펙 공백을 사람에게 보고한다.

### 12.3 스펙 ↔ 구현 추적 매트릭스 (권장 산출물)
보정을 쉽게 하려고, 구현 시 아래 매핑을 유지한다(주석/표 어디든).

| 스펙 ID | 구현 위치 | 검증 테스트 |
|---------|-----------|-------------|
| INV-4 (금액 원장 보존) | `domain/transaction.ts` | `TC-A24` |
| REQ-A6 (최소개수 잔돈) | `domain/change.ts` | `TC-A09` |
| A-7.1 (프로모 할인) | `domain/pricing.ts` | `TC-A16, TC-A17` |
| A-7.2 (동적 가격) | `domain/pricing.ts` | `TC-A18, TC-A19` |
| A-7.3 (배출 실패) | `domain/dispense.ts` | `TC-A20, TC-A21` |
| … | … | … |

### 12.4 핵심 원칙
- **감지 없는 진행 금지**: 게이트를 건너뛰고 "완료"라 하지 않는다(R8).
- **추측 없는 보정**: 스펙 공백은 코드가 아니라 문서에서 먼저 메운다(R1·R7).
- **되돌릴 수 있게**: 작은 단위(R6) + 회귀 테스트(§12.2-4)로 언제든 스펙 기준선으로 복귀 가능하게 유지한다.
